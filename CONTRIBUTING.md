# Contributing Guide

このリポジトリへの関心に感謝する。  
本ドキュメントは、提案・議論・変更を行う際の基本方針を示す。

本プロジェクトは **個人学習・設計検証を主目的**としており、  
完成物の速さや網羅性よりも、**設計の一貫性と理解可能性**を重視する。

---

## 基本方針

- 提案・議論は歓迎する
- 設計判断は慎重に扱う
- 変更は段階的・説明可能であることを重視する

最終的な判断・マージ権限は、  
リポジトリオーナーにある。

---

## 事前に読んでほしいドキュメント

変更や提案を行う前に、以下のドキュメントを参照してほしい。

- [`architecture.md`](./architecture.md)  
  変更してはならない構造・依存方向・責務分離

- [`design.md`](./design.md)  
  現在の設計判断・仮説・試行錯誤のログ

- [`copilot-instructions.md`](./copilot-instructions.md)  
  本リポジトリで重視している価値観・制約

これらと矛盾する変更は、  
**設計変更**として扱われる。

---

## Issue について

以下の内容は Issue としての投稿を歓迎する。

- 設計に関する疑問・質問
- 改善案や代替設計の提案
- 不具合や挙動に関する報告
- 学習・理解のための質問

Issue では、  
**結論よりも前提・意図・観点**を重視する。

---

## Pull Request（PR）について

### 歓迎される PR

- バグ修正
- テストの追加・改善
- 可読性・理解性の向上
- 既存設計の範囲内での改善

### 注意が必要な PR

以下を含む PR は、  
事前に Issue での議論を推奨する。

- レイヤー責務の変更
- 公開 API（pub）の意味的変更
- 依存関係の追加・変更
- データモデルの根本的な変更

これらは **設計変更** に該当する可能性がある。

---

## PR 作成時のお願い

- 変更の **目的** と **理由** を明確に記載する
- 差分は可能な限り小さく保つ
- 設計判断に関わる場合は、代替案も簡単に触れる
- コードだけでなく、必要に応じてドキュメントも更新する

---

## コーディング・設計に関する指針（要約）

詳細は各設計ドキュメントを参照してほしいが、  
要点は以下の通り。

- 正しさ > 速さ > 短さ
- レイヤー境界を破らない（runtime → logic → core）
- 公開 API には rustdoc と `# Examples` を原則付ける
- コメントは「なぜ」を書く

---

## 最後に

このリポジトリは、  
**設計を考え続けるための実験場**である。

提案・議論・問いかけは歓迎するが、  
拙速な結論よりも、  
**理解可能な過程**を大切にしたい。

その点を共有できるのであれば、  
あなたの関与はとても歓迎される。
